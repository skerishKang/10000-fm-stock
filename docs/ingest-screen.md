# Ingest 화면 설계 — 상세 명세서

> **버전:** 1.0  
> **연관 이슈:** #5  
> **목표:** 외부 자료(유튜브 영상, 리포트 PDF)를 내부 `source`와 `segment` 후보로 변환하는 입력 화면 설계

---

## 1. 배경

이 프로젝트의 첫 번째 진짜 기능은 **자료 입력 화면(Ingest Screen)**이다.  
좋은 데이터를 넣지 못하면 이후 모든 페이지(Review, Dashboard, Search)가 의미를 잃는다.  
Ingest 화면은 사용자가 접하는 첫 번째 인터랙션 지점이며, 외부의 원자료(raw data)를
시스템이 이해할 수 있는 구조화된 형태로 바꾸는 관문 역할을 한다.

**핵심 원칙:**

- 입력은 가볍게, 저장은 정확하게
- 원문(raw text/full PDF)은 절대 저장하지 않음 — 참조 정보(제목, 발행자, URL, 구간)만 저장
- AI(LLM)가 없어도 전체 기능이 동작해야 함
- 저장 결과는 JSON으로 표현 가능하며, 이후 Review 화면에서 확인/수정됨

---

## 2. 용어 정의

| 용어 | 설명 |
|------|------|
| **Source** | 하나의 원자료 단위. 유튜브 영상 하나 또는 리포트 PDF 하나가 각각 하나의 source가 된다. |
| **Segment** | source 내에서 사용자가 지정한 구간. 유튜브는 시간 구간, 리포트는 페이지 구간. 여러 segment가 하나의 source에 속할 수 있다. |
| **Candidate** | 아직 claim 또는 knowledge_note로 확정되지 않은 상태. Ingest 화면에서 생성되는 모든 segment는 candidate 상태이다. |
| **Review** | Ingest 이후 다음 단계 화면. candidate를 검토하고 claim/knowledge_note로 승격하거나 폐기한다. |

---

## 3. 화면 개요

Ingest 화면은 크게 두 가지 입력 모드를 제공한다.

1. **유튜브 입력 모드** — YouTube 영상에서 특정 구간을 발췌하여 저장
2. **리포트 입력 모드** — 리포트 PDF의 특정 페이지/문단을 발췌하여 저장

두 모드는 상단 탭 또는 토글 버튼으로 전환한다.  
기본값은 유튜브 모드이다.

각 모드별 입력 필드는 아래 상세 섹션에서 설명한다.

---

## 4. 유튜브 입력 모드 — 상세

### 4.1 입력 필드 목록

| # | 필드명 | 타입 | 필수 | 설명 |
|---|--------|------|------|------|
| 1 | 영상 URL | `string` | ✅ | YouTube 영상의 전체 URL (https://youtube.com/watch?v=...) |
| 2 | 영상 제목 | `string` | ✅ | 영상의 제목, 자동 채우기 권장 |
| 3 | 채널명 / 발행자 | `string` | ✅ | 영상을 업로드한 채널 이름 |
| 4 | 발행일 | `date` (YYYY-MM-DD) | ✅ | 영상이 업로드된 날짜 |
| 5 | 시작 시간 | `time` | ✅ | 발췌할 구간의 시작 시간 |
| 6 | 끝 시간 | `time` | ✅ | 발췌할 구간의 끝 시간 |
| 7 | 발언자 이름 | `string` | ❌ | 해당 구간에서 발언 중인 사람 (여럿일 경우 쉼표 구분) |
| 8 | 관련 종목 | `string[]` | ❌ | 해당 구간과 관련된 종목 코드 (예: 삼성전자, 005930) |
| 9 | 관련 산업 | `string[]` | ❌ | 해당 구간과 관련된 산업 분류 (예: 반도체, 2차전지) |
| 10 | 구간 메모 | `string` | ❌ | 해당 구간에 대한 간단한 메모 또는 태그 |
| 11 | 자막 / 본문 텍스트 | `text` | ❌ | 자막이 있으면 자막, 없으면 사용자가 직접 적은 텍스트 |

### 4.2 시간 입력 형식

시작 시간과 끝 시간은 아래 형식을 모두 지원한다.  
내부 저장은 **항상 초(second) 단위 정수**로 통일한다.

| 입력 예시 | 변환 결과 | 설명 |
|-----------|-----------|------|
| `312` | 312초 | 숫자만 입력 → 그대로 초 |
| `5:12` | 312초 | MM:SS 형식 |
| `1:05:12` | 3912초 | H:MM:SS 형식 |
| `01:05:12` | 3912초 | HH:MM:SS 형식 (0 패딩 허용) |
| `0:30` | 30초 | 30초 구간 |

**변환 로직 (의사코드):**

```
function parseTime(input):
    input = input.trim()
    if input is empty: return None
    if input matches /^\d+$/:
        return int(input)  # 숫자만 → 초
    if input matches /^(\d+):(\d+)$/:
        minutes = group1, seconds = group2
        return minutes * 60 + seconds
    if input matches /^(\d+):(\d+):(\d+)$/:
        hours = group1, minutes = group2, seconds = group3
        return hours * 3600 + minutes * 60 + seconds
    raise FormatError("지원하지 않는 시간 형식입니다")
```

### 4.3 유효성 검사

- 시작 시간 < 끝 시간 이어야 함
- 끝 시간이 영상 전체 길이를 초과할 수 없음 (선택적, API 연동 시)
- 영상 URL은 `youtube.com` 또는 `youtu.be` 도메인이어야 함
- 발행일은 미래 날짜일 수 없음

## 5. 리포트 입력 모드 — 상세

### 5.1 입력 필드 목록

| # | 필드명 | 타입 | 필수 | 설명 |
|---|--------|------|------|------|
| 1 | 리포트 제목 | `string` | ✅ | 리포트의 전체 제목 |
| 2 | 발행사 | `string` | ✅ | 리포트를 발행한 증권사/리서치 기관 |
| 3 | 애널리스트 이름 | `string` | ❌ | 리포트를 작성한 애널리스트 (여럿일 경우 쉼표 구분) |
| 4 | 발행일 | `date` (YYYY-MM-DD) | ✅ | 리포트가 발행된 날짜 |
| 5 | 공개 URL | `string` | ❌ | 리포트를 공개적으로 볼 수 있는 URL (증권사 사이트 등) |
| 6 | 개인용 privatePath | `string` | ❌ | 사용자가 로컬에 보관 중인 파일 경로 (예: `D:\reports\2025\...`) |
| 7 | 페이지 번호 | `string` | ❌ | 발췌할 페이지 번호. 범위 지정 가능 (예: `3` 또는 `3-5` 또는 `3,4,5`) |
| 8 | 섹션 제목 | `string` | ❌ | 해당 페이지/문단이 속한 섹션 제목 (예: "Valuation", "Risk Factors") |
| 9 | 관련 종목 | `string[]` | ❌ | 해당 페이지와 관련된 종목 코드 |
| 10 | 관련 산업 | `string[]` | ❌ | 해당 페이지와 관련된 산업 분류 |
| 11 | 페이지/문단 메모 | `string` | ❌ | 해당 페이지에 대한 간단한 메모 |
| 12 | 사용자 본문 일부 | `text` | ❌ | 사용자가 직접 붙여넣은 리포트 본문 일부 (전문 금지) |

### 5.2 페이지 번호 저장 방식

페이지 번호 필드는 아래 형식을 모두 지원한다.

| 입력 예시 | 내부 저장 (정규화) | 설명 |
|-----------|-------------------|------|
| `3` | `[3]` | 단일 페이지 |
| `3-5` | `[3, 4, 5]` | 범위 지정 |
| `3,5,7` | `[3, 5, 7]` | 개별 페이지 목록 (쉼표 구분) |
| `3-5,7,9-11` | `[3, 4, 5, 7, 9, 10, 11]` | 혼합 형식 |
| `3, 5, 7` | `[3, 5, 7]` | 공백 허용 |

### 5.3 중요: 원문 저장 금지

**리포트 원문(PDF 전체, 전체 텍스트)을 데이터베이스에 저장해서는 안 된다.**  
이는 저작권 및 법적 문제를 방지하기 위함이다.

저장 가능한 항목:
- 메타데이터 (제목, 발행사, 발행일 등)
- 사용자가 직접 작성한 메모
- 사용자가 직접 붙여넣은 **일부** 텍스트 (전문이 아닌 인용 수준)
- 공개 URL 및 로컬 파일 경로 (참조 용도)


### 6.3 공통 필드 설명

| 필드 | 설명 |
|------|------|
| `source.type` | `"youtube"` 또는 `"report"` |
| `source.language` | ISO 639-1 언어 코드 (ko, en, ja 등) |
| `source.createdAt` | Ingest 화면에서 저장한 시각 (ISO 8601) |
| `source.tags` | 사용자 또는 AI가 자동 추천한 태그 목록 |
| `segment.sourceId` | 부모 source의 UUID |
| `segment.status` | 항상 `"candidate"` — 아직 claim/knowledge_note로 확정되지 않음 |
| `segment.textContent` | 자막/본문 또는 사용자 입력 텍스트. 리포트의 경우 **전문이 아닌 일부만** |
| `segment.*Display` | 사용자가 입력한 원본 형식을 보존하는 display 전용 필드 |

## 7. 저장 후 액션 버튼

모든 필드가 입력되고 유효성 검사를 통과하면, 사용자는 아래 세 가지 버튼 중 하나를 선택한다.

### 7.1 저장만 하기

- 입력된 데이터를 그대로 데이터베이스에 저장
- 화면 초기화, 다음 입력 준비
- 저장 완료 토스트 메시지 표시 (예: "Segment가 저장되었습니다 (candidate 상태)")

### 7.2 Review 화면으로 이동하기

- 저장 후 자동으로 Review 화면으로 이동
- Review 화면은 방금 저장한 segment를 최상단에 표시
- 사용자가 바로 검토/수정/승격 작업을 이어갈 수 있음

### 7.3 Claim / Knowledge 후보 만들기

- 저장 후 AI(LLM)를 호출하여 segment의 textContent를 분석
- AI가 추출한 claim과 knowledge_note 후보를 생성
- 생성된 후보는 Review 화면에서 사용자가 확정/수정/폐기
- **AI가 없을 경우 이 버튼은 비활성화**되며, 사용자는 Review 화면에서 수동 작성

### 7.4 버튼 동작 비교

| 버튼 | 저장 수행 | 화면 전환 | AI 호출 |
|------|----------|-----------|---------|
| 저장만 하기 | ✅ | ❌ (현재 화면 유지) | ❌ |
| Review로 이동 | ✅ | ✅ → Review | ❌ |
| Claim/Knowledge 후보 | ✅ | ✅ → Review | ✅ |

---

## 8. Review 화면 연결 구조

Ingest 화면에서 저장한 segment는 다음 단계인 **Review 화면**에서 처리된다.

```
[Ingest Screen] --(저장)--> [Database: source + segment (candidate)]
                                  |
                                  v
[Review Screen] <--(불러오기)----+
     |
     ├── Claim으로 승격 (is_claim: true, is_knowledge_note: false)
     ├── Knowledge Note로 승격 (is_claim: false, is_knowledge_note: true)
     ├── 수정 후 승격
     ├── 보류 (status 유지)
     └── 폐기 (삭제)
```

**Ingest 화면은 오직 candidate 생성까지만 담당한다.**  
candidate를 최종 확정(claim/knowledge_note)하거나 폐기하는 것은 Review 화면의 책임이다.

## 9. AI 자동 추출 (선택 사양)

AI가 연결되어 있을 경우 Ingest 화면에서 아래 기능이 추가로 동작한다.

### 9.1 저장 시 자동 제안

- 영상 URL 입력 시 자동으로 제목/채널명/발행일 추천 (YouTube API 활용)
- 텍스트 입력 시 관련 종목/산업 자동 태깅 제안
- 텍스트 요약을 통한 `summary` 필드 자동 생성

### 9.2 Claim/Knowledge 후보 생성

- "Claim/Knowledge 후보 만들기" 버튼 동작 시
- segment.textContent를 LLM에 전달
- LLM이 반환한 JSON을 파싱하여 후보 목록 생성
- 각 후보는 Review 화면에서 사용자 승인 필요

### 9.3 AI 미연결 시 동작

- 모든 AI 의존 기능이 비활성화됨
- 사용자가 모든 필드를 수동 입력
- "Claim/Knowledge 후보 만들기" 버튼 숨김 또는 비활성화
- Review 화면에서 수동으로 claim/knowledge_note 작성 가능

## 10. 시간 변환 로직 — 상세 구현

### 10.1 정규식 기반 파싱

```
입력값: "312"       → /^\d+$/       → 312 (초)
입력값: "5:12"      → /^(\d+):(\d+)$/ → 5*60 + 12 = 312 (초)
입력값: "1:05:12"   → /^(\d+):(\d+):(\d+)$/ → 1*3600 + 5*60 + 12 = 3912 (초)
입력값: "01:05:12"  → /^(\d+):(\d+):(\d+)$/ → 1*3600 + 5*60 + 12 = 3912 (초)
```

### 10.2 우선순위

1. 숫자만 → 초 단위 그대로 해석
2. `:`가 1개 → MM:SS
3. `:`가 2개 → H:MM:SS
4. 그 외 → 오류 반환

### 10.3 예외 처리

- 음수 입력 → 오류
- 0초 입력 → 오류 (구간이 없음)
- 시작 시간 ≥ 끝 시간 → 오류
- 3자리 이상 숫자+혼합 (예: `123:45`) → H:MM:SS로 해석하지 않고 MM:SS로 해석
  - `123:45` → 123*60 + 45 = 7425초 (약 2시간)

## 11. 페이지 번호 변환 로직 — 상세 구현

### 11.1 정규식 기반 파싱

리포트 모드의 페이지 번호는 다음 순서로 파싱한다.

1. 입력 문자열을 `,`로 분할
2. 각 토큰에 대해:
   - `\d+-\d+` 패턴 → 범위로 확장 (예: `3-5` → `[3,4,5]`)
   - `\d+` 패턴 → 단일 페이지 (예: `3` → `[3]`)
3. 중복 제거 및 오름차순 정렬

### 11.2 예시

```
입력: "3"         → [3]
입력: "3-5"       → [3, 4, 5]
입력: "1,3,5"     → [1, 3, 5]
입력: "3-5,7,9-11" → [3, 4, 5, 7, 9, 10, 11]
입력: "3, 5, 7"   → [3, 5, 7] (공백 무시)
```
