# Review 화면 설계

## 1. 개요

본 문서는 Claim/Knowledge 후보 검수(Review) 화면의 설계 명세를 기술한다. 검수자는 이 화면을 통해 AI가 추출한 Claim 및 Knowledge Note 후보를 검토하고, 채택·수정·삭제 등의 판정을 내린다.

### 1.1 목적

- AI 추출 결과물의 정확성과 신뢰성을 사람이 검증
- 잘못 추출된 정보(종목명 혼동, 목표가 오인, 발언자 오연결 등)를 걸러냄
- 검수 완료된 양질의 데이터만 메인 DB로 승격
- 사람의 판단이 개입된 Ground Truth 데이터 구축

### 1.2 대상 사용자

- 금융 데이터 검수자 (애널리스트, 리서치 어시스턴트)
- 데이터 품질 관리자
- 시스템 운영자 (문제 발생 시 원인 파악 용도)

### 1.3 화면 접근 경로

- 대시보드 → 검수 대기 목록 → 개별 후보 검수 화면
- 또는 검수 대기 알림 → 개별 후보 검수 화면

## 2. Claim 후보 카드

### 2.1 개요

Claim(발언/주장) 단위의 검수 카드. 하나의 발언이 하나의 Claim 카드로 표현된다.

### 2.2 표시 필드

| 필드 | 설명 | 데이터 타입 | 예시 |
|------|------|-----------|------|
| 발언자 | 해당 발언을 한 사람 | string | "김OO 애널리스트" |
| 출처 | 발언이 포함된 원본 자료 | string | "2024-03-15 삼성전자 리포트" |
| 구간/페이지 | 출처 내 위치 정보 | string | "p.3, 두 번째 문단" |
| 종목명 | 관련 종목 이름 | string | "삼성전자" |
| 티커 | 종목 티커 심볼 | string | "005930" |
| 산업 | 종목 산업 분류 | string | "반도체/IT" |
| 방향성 | 발언의 전망 방향 | enum | "긍정/부정/중립" |
| 발언내용 | 실제 발언 텍스트 | text | "삼성전자의 HBM3E 양산은 2분기 내..." |
| 근거 | 발언의 근거/이유 | text | "HBM3E 시장 점유율 확대 전망" |
| 기준일 | 발언 기준 시점 | date | "2024-03-15" |
| 기준가 | 기준 시점의 가격 | number | 73,500 |
| 목표일 | 전망 목표 시점 | date | "2024-06-30" |
| 목표가 | 전망 목표 가격 | number | 85,000 |
| 상태 | 현재 검수 상태 | enum | "대기/채택/수정후채택/삭제/교육용전환/나중에검토" |

### 2.3 동작 버튼

검수자는 아래 5가지 동작 중 하나를 선택한다:

1. **채택 (Accept)**
   - AI 추출 내용을 그대로 신뢰할 수 있을 때
   - 모든 필드가 정확하고 추가 수정이 불필요한 경우
   - 클릭 시 상태가 "채택"으로 변경, 메인 DB로 이동

2. **수정후채택 (Accept with Edit)**
   - 일부 필드가 부정확하여 수정이 필요한 경우
   - 클릭 시 인라인 편집 모드로 전환
   - 수정 완료 후 저장하면 상태가 "수정후채택"으로 변경
   - 변경 이력이 자동 기록됨

3. **삭제 (Delete)**
   - AI가 잘못 추출한 경우 (예: 관련 없는 내용을 Claim으로 오인)
   - 발언 자체가 의미 없는 경우
   - 클릭 시 확인 다이얼로그 후 삭제

4. **교육용전환 (Convert to Training)**
   - 잘못 추출되었으나 AI 교육 데이터로 가치가 있는 경우
   - 오류 패턴 라벨링과 함께 교육 데이터셋으로 이동
   - 이후 AI 모델 개선에 활용

5. **나중에검토 (Review Later)**
   - 현재 판단하기 어렵거나 추가 정보가 필요할 때
   - "보류" 상태로 저장, 나중에 다시 검토 가능

## 3. Knowledge Note 후보 카드

### 3.1 개요

Knowledge Note(지식 노트) 단위의 검수 카드. Claim들을 종합하여 도메인 지식으로 정리한 단위다.

### 3.2 표시 필드

| 필드 | 설명 | 데이터 타입 | 예시 |
|------|------|-----------|------|
| 산업 | 대상 산업 분류 | string | "반도체" |
| 종목 | 관련 종목명 | string | "삼성전자" |
| 주제 | 지식 노트의 주제 | string | "HBM 시장 동향 및 경쟁력 분석" |
| 난이도 | 내용 난이도 레벨 | enum | "초급/중급/고급" |
| 요약 | 노트의 핵심 요약 | text | "삼성전자는 HBM3E 양산을 통해..." |
| 핵심포인트 | 핵심 포인트 리스트 | list | ["HBM3E 양산 시점 2분기", "점유율 확대 전망"] |
| 태그 | 관련 태그 | list | ["HBM", "HBM3E", "삼성전자", "메모리"] |
| 연결 source | 참조한 Source ID | string | "SRC-20240315-SAMSUNG" |
| 연결 segment | 참조 Segment ID | string | "SEG-003" |
| 연결 expert | 참조 Expert ID | string | "EXP-KIM-001" |

### 3.3 동작 버튼

1. **채택 (Accept)**
2. **수정후채택 (Accept with Edit)**
3. **삭제 (Delete)**
4. **Claim으로전환 (Convert to Claim)**
   - Knowledge Note 수준이 아닌, 단일 Claim으로 관리해야 할 경우
   - Note → Claim으로 다운그레이드하여 관리
5. **나중에검토 (Review Later)**

## 5. AI 실수 패턴

검수 시 주의해야 할 AI의 대표적인 실수 패턴을 정리한다.

### 5.1 종목명 혼동

- **증상**: 유사한 이름의 종목을 혼동 (예: "삼성전자"와 "삼성SDI" 혼동, "셀트리온"과 "셀트리온헬스케어" 혼동)
- **원인**: 종목명 임베딩 유사도가 높을 때 발생
- **검수 포인트**: 종목명과 티커가 일치하는지 반드시 확인
- **예시**: "LG에너지솔루션" 발언을 "LG화학" Claim으로 추출

### 5.2 목표가/현재가 혼동

- **증상**: 발언 내 현재가와 목표가를 뒤바꿔 추출
- **원인**: "현재 7만원 수준이나 8만원까지 상승 가능" 등의 문장에서 수치 해석 오류
- **검수 포인트**: 기준가 < 목표가 (긍정), 기준가 > 목표가 (부정) 인지 검증
- **예시**: 기준가 85,000원, 목표가 73,500원으로 추출 (역전)

### 5.3 가능성 표현 → 매수 오해

- **증상**: "가능성이 있다"는 표현을 "매수 추천"으로 오해석
- **원인**: AI가 확률 표현과 투자 의견을 구분하지 못함
- **검수 포인트**: 발언내용에 "가능성", "전망", "예상" 등의 단어가 있으면 방향성을 재확인
- **예시**: "하반기 실적 개선 가능성" → "긍정" 방향성은 맞으나, "매수 추천"으로 확대 해석하지 않도록 주의

### 5.4 발언자 오연결

- **증상**: 잘못된 사람에게 발언이 귀속됨
- **원인**: 리포트 내 인용구와 작성자를 잘못 매핑
- **검수 포인트**: 발언자와 출처의 관계 검증 (리포트 작성자 ≠ 인용된 외부 발언자일 수 있음)
- **예시**: "CEO가 '시장 전망이 밝다'고 언급" → 리포트 작성자 애널리스트가 CEO 발언을 인용한 건데, AI가 애널리스트의 발언으로 추출

### 5.5 구간/페이지 위치 오류

- **증상**: 발언이 위치한 페이지나 문단이 실제와 다름
- **원인**: PDF 파싱 시 페이지 구분 오류 또는 문장 분할 오류
- **검수 포인터**: 원본 출처와 위치를 교차 검증

### 5.6 방향성 오류

- **증상**: 긍정/부정/중립 방향성을 잘못 판단
- **원인**: 이중 부정, 조건부 긍정, 맥락 무시
- **검수 포인트**: "부정적이지 않다" = 긍정, "다만 리스크도 존재" = 혼합(중립) 등
- **예시**: "실적 부진 가능성은 낮다" → AI가 "부진"에 주목하여 "부정"으로 분류

### 5.7 산업 분류 오류

- **증상**: 종목의 산업을 잘못 분류
- **원인**: 사업 다각화 기업의 경우 AI가 주요 사업을 오인
- **검수 포인트**: 해당 발언의 맥락에 맞는 산업 분류인지 확인

## 6. 사람 검수 원칙

### 6.1 기본 원칙

1. **AI 결과물을 맹신하지 않는다**
   - AI 추출 결과는 참고용일 뿐, 최종 판단은 사람이 내린다
   - AI의 정확도가 95%라도, 검수자는 5%의 오류를 찾는 것이 목적

2. **원본 출처를 항상 확인한다**
   - 발언내용만 보고 판단하지 말고, 가능하면 원본 PDF/영상에서 해당 부분을 직접 확인
   - 구간/페이지 정보를 활용하여 원본과 대조

3. **일관된 기준을 적용한다**
   - 유사한 케이스는 유사한 판정을 내려야 함
   - 검수자 간 편차를 줄이기 위해 정기적인 캘리브레이션 필요

### 6.2 판정 기준 가이드라인

| 조건 | 권장 동작 |
|------|---------|
| 모든 필드가 정확함 | 채택 |
| 1-2개 필드만 부정확하고 수정 가능 | 수정후채택 |
| 3개 이상 필드 부정확 또는 핵심 내용 오류 | 삭제 또는 교육용전환 |
| Claim으로 관리하는 것이 적절하지 않음 (Knowledge Note의 경우) | Claim으로전환 |
| 판단을 위한 정보가 부족함 | 나중에검토 |

### 6.3 품질 기준

- **채택**: 모든 필드가 원본과 100% 일치
- **수정후채택**: 수정 후 모든 필드가 원본과 100% 일치, 수정 이력 보존
- **삭제**: 명백한 AI 오추출 사례
- **교육용전환**: 오류 패턴에 레이블링 가능한 사례

### 6.4 검수 속도 목표

- Claim 카드: 건당 30초 ~ 2분
- Knowledge Note 카드: 건당 1분 ~ 5분
- 1세션 최대 연속 검수: 50건 (이후 휴식 권장)

## 7. AI 없이 수동 동작

### 7.1 개요

검수 화면은 AI 없이도 완전히 수동으로 동작 가능해야 한다. AI 추출 결과는 어시스트 역할일 뿐, 시스템의 핵심 기능은 사람이 직접 데이터를 입력하고 관리할 수 있어야 한다.

### 7.2 수동 Claim 등록

```json
{
  "type": "claim",
  "data": {
    "speaker": "",
    "source": "",
    "location": {},
    "stock": {"name": "", "ticker": "", "industry": ""},
    "direction": "",
    "content": "",
    "rationale": "",
    "reference_date": null,
    "reference_price": null,
    "target_date": null,
    "target_price": null,
    "status": "manual_input"
  }
}
```

- 모든 필드를 빈 값으로 시작하여 검수자가 직접 입력
- AI 추출 없이도 동일한 UI/UX로 데이터 등록 가능
- 수동 등록 데이터는 AI 추출 데이터와 동일한 검수 프로세스 진행

### 7.3 수동 Knowledge Note 등록

- Claim을 선택하여 Knowledge Note로 수동 그룹핑 가능
- 또는 빈 Knowledge Note를 생성하여 직접 모든 필드 작성
- 연결 정보(source/segment/expert)는 수동 선택 UI 제공

### 7.4 오프라인 검수 지원

- 검수 데이터를 JSON/CSV로 내보내기 가능
- 엑셀에서 일괄 검수 후 다시 시스템으로 가져오기 가능
- 배치(Batch) 검수 모드 지원 (일괄 채택/삭제)

### 7.5 키보드 단축키 (수동 작업 효율화)

| 단축키 | 동작 |
|--------|------|
| `A` | 채택 |
| `E` | 수정후채택 |
| `D` | 삭제 |
| `T` | 교육용전환 |
| `C` | Claim으로전환 (Knowledge Note에서) |
| `R` | 나중에검토 |
| `N` | 다음 카드 |
| `P` | 이전 카드 |
| `Ctrl+S` | 수정사항 저장 |
| `Esc` | 수정 모드 취소 / 다이얼로그 닫기 |

## 8. 데이터 모델

### 8.1 Review Candidate (검수 후보)

```
ReviewCandidate {
  id: string (PK)
  type: enum ("claim" | "knowledge_note")
  source_system: string ("ai_extraction" | "manual_input")
  data: jsonb (Claim 또는 Knowledge Note 데이터)
  status: enum ("pending" | "accepted" | "accepted_edited" | "deleted" | "training" | "converted" | "later")
  ai_confidence: float (0.0 ~ 1.0)
  created_at: timestamp
  updated_at: timestamp
}
```

### 8.2 Review Action (검수 행위)

```
ReviewAction {
  id: string (PK)
  candidate_id: string (FK → ReviewCandidate)
  reviewer: string
  action: enum ("accept" | "accept_with_edit" | "delete" | "convert_to_training" | "convert_to_claim" | "review_later")
  modifications: jsonb (null if no edit)
  comment: text
  previous_status: enum
  new_status: enum
  reviewed_at: timestamp
}
```

### 8.3 UI State (화면 상태)

```
ReviewUIState {
  current_index: int
  total_count: int
  filter: {
    type: string | null,
    status: string | null,
    reviewer: string | null,
    date_range: [date, date] | null,
    ai_confidence_min: float | null
  }
  is_editing: boolean
  has_unsaved_changes: boolean
}
```

## 10. 향후 개선 사항

1. **AI 추천 순서**: AI 신뢰도(AI Confidence) 순으로 후보 정렬하여 효율적인 검수 지원
2. **유사 사례 그룹핑**: 동일 종목/산업의 유사 후보를 그룹으로 검수
3. **자동 수정 제안**: AI 실수 패턴 감지 시 자동 수정안 제안
4. **검수자 간 교차 검증**: 중요 Claim은 2인 이상 검수자 확인
5. **대시보드 통계**: 검수 속도, 정확도, 잔여 건수 등을 대시보드로 제공
6. **피드백 루프**: 검수 결과를 AI 모델 재학습 피드백으로 자동 전송
